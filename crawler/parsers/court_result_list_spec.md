# Court Result List Probe Spec (Day 4)

## input artifact
- `data/raw/court_probe/search_attempt_last_30_days.html`
- Generated by Day 3 POST form submission probe.

## parsing hypothesis
- The response may be one of two classes:
  1. A real result list page containing repeated judgment entries.
  2. The same search form page returned again (e.g., invalid payload, required field missing, or server fallback).
- Result entries may appear as any repeated structure (`tr`, `li`, `div`, `section`, `article`) rather than a fixed selector.

## observed repeated structures
- Probe logic scans all repeated signature groups by tag/class combination.
- Candidate repeated structures include:
  - table-like rows (`tr`)
  - list-like containers (`li`)
  - panel/card-like containers (`div`/`section`/`article`)
- Actual observed counts are written to:
  - `data/parsed/court_probe/result_items_probe_report.txt`

## extracted fields
Each extracted result item attempts the following fields:
- `raw_text`
- `title`
- `case_number`
- `judgment_date`
- `court`
- `detail_url`
- `document_url`

Output file:
- `data/parsed/court_probe/result_items_last_30_days.json`

## unknowns
- Whether case-number format is fully covered by current regex hypotheses.
- Whether date text in list items uses locale-only formats that need stronger normalization.
- Whether document links are embedded by JavaScript event handlers rather than plain `<a href>`.
- Whether current POST payload already reaches a valid result listing state.

## whether current HTML is enough for result parsing
- Decision is made by probe signals:
  - repeated blocks found
  - case/date hits in extracted items
  - page-level heuristic for form markers vs result markers
- Refer to report booleans for concrete status:
  - `whether page looks like real result page`
  - `whether page looks like same search form page returned again`

## recommended next step
- If report says **real result page = true** and extraction count is meaningful:
  - next step: parse detail pages.
- If report says **search form returned again = true** or extraction count is zero:
  - next step: refine POST payload (hidden fields, required filters, locale/state parameters).
- If repeated evidence remains weak after payload refinement:
  - escalate to Playwright as a controlled fallback (not in this Day 4 implementation).
